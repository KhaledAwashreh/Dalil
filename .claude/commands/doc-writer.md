# Dalil Doc Writer

Sub-agent that produces documentation grounded in Dalil's knowledge base.
Every claim in the document is backed by a retrieved case — no hallucinated facts.

## Variables

- `TOPIC`: `$ARGUMENTS` with flags stripped — what the document is about (required)
- `VAULT`: value of `--vault` flag, default `default`
- `DOC_TYPE`: value of `--type` flag, one of `spec`, `summary`, `architecture`, `runbook`, `analysis`, default `summary`
- `OUTPUT_PATH`: value of `--output` flag, default prints to stdout
- `DALIL_URL`: env var `DALIL_URL`, default `http://localhost:8000`

## Rules

- Every factual claim MUST trace back to a Dalil case — include inline references
- DO NOT invent facts, statistics, or case details
- If Dalil's knowledge is insufficient for a section, mark it as `[NEEDS INPUT]` rather than guessing
- DO include a sources section at the end with case IDs and titles
- DO include all request_ids for feedback

---

## Workflow

### Phase 1 — Gather Context

Launch the `dalil-context-gatherer` sub-agent to collect comprehensive context:

```
Prompt: /dalil-context-gatherer "${TOPIC}" --vault=${VAULT} --depth=deep
```

Parse the context package. If `VAULT_EMPTY`, stop and tell the caller to ingest knowledge first.

### Phase 2 — Plan Document Structure

Based on `DOC_TYPE`, plan the document sections:

**spec:**
- Overview, Problem Statement, Requirements, Constraints, Prior Art, Risks, Open Questions

**summary:**
- Executive Summary, Key Findings, Supporting Evidence, Recommendations

**architecture:**
- System Overview, Components, Data Flow, Integration Points, Decisions & Trade-offs, Known Risks

**runbook:**
- Purpose, Prerequisites, Procedures, Troubleshooting, Escalation, Related Incidents

**analysis:**
- Context, Findings, Patterns, Contradictions, Recommendations, Confidence Assessment

### Phase 2.5 — Entity Graph for Coverage

Use the entity graph to ensure comprehensive topic coverage:

```bash
dalil_url="${DALIL_URL:-http://localhost:8000}"
curl -sf "${dalil_url}/vault/entities/${TOPIC_ENTITY}?vault=${VAULT}"
```

Use entity relationships to identify sections or angles that the initial context gathering may have missed.
Use `/traverse` with `start_id` and `max_depth` for relationship mapping between key concepts.

### Phase 3 — Write Document

For each section:

1. Identify which cases from the context package are relevant to this section
2. Write the section content, citing cases inline as `[Case: {title}]`
3. If no cases support a section, write `[NEEDS INPUT] — no relevant cases found for this section`
4. Include confidence indicators using MuninnDB Bayesian scores and score breakdowns:
   - High confidence (score > 0.7, multiple supporting cases) — show score breakdown if available
   - Medium confidence (score 0.5-0.7, single case)
   - Low confidence (score < 0.5) — flag explicitly with breakdown details

### Phase 4 — Add Metadata

Append to the document:

```markdown
---

## Document Metadata

**Generated by:** Dalil Doc Writer
**Topic:** {TOPIC}
**Vault:** {VAULT}
**Type:** {DOC_TYPE}
**Cases Referenced:** {count}
**Confidence:** {overall assessment}

### Sources

| Case | Title | Type | Score |
|------|-------|------|-------|
| {id} | {title} | {type} | {score} |

### Gaps

{Sections marked as [NEEDS INPUT] — areas where the vault lacks knowledge}

### Feedback

To improve future documents, provide feedback on the consultations used:
{list of request_ids with feedback commands}
```

### Phase 5 — Output

If `OUTPUT_PATH` is set, write the document to that path using the Write tool.
Otherwise, print the full document to stdout.
